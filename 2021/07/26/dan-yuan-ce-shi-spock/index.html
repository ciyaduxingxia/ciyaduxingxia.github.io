单元测试-spock

一、spock是什么

spock是国外一款优秀的测试框架，能够让我们的测试代码规范化，结构层次清晰，结合groovy动态语言的特点以及自身提供的各种标签让编写测试代码更加高效和简洁，提供一种通用的，简单的，结构化的描述语言。

简单说Spock的特点如下:

  • 让我们的测试代码更规范, 内置多种标签来规范单测代码的语义, 从而让我们的测试代码结构清晰, 更具可读性, 降低后期维护难度
  • 提供很多实用的标签, 比如: where, with, thrown… 帮组我们应对复杂的测试场景
  • 再加上使用groovy这种动态语言来编写测试代码, 可以让我们编写的测试代码更简洁, 适合敏捷开发, 提高编写单测代码的效率
  • 遵从BDD行为驱动开发模式, 不单是为了测试覆盖率而测试, 有助于提升代码质量
  • IDE兼容性好,自带mock功能

二、spock标签及符号的含义

Spock标签的作用：

  • given: 输入条件(前置参数)
  • when: 执行行为(mock接口、真实调用)
  • then: 输出条件(验证结果)
  • and: 衔接上个标签，补充的作用
  • expect: 相当于when和then的结合

then模块作用是验证被测方法的结果是否正确，符合预期值，所以这个模块里的语句必须是boolean表达式，类似于junit的assert断言机制，但你不必显示的写assert，这也是一种约定优于配置的思想
then块中使用了spock的with功能，可以验证返回结果response对象内部的多个属性是否符合预期值，这个相对于junit的assertNotNull或assertEquals的方式更简单一些

where模块第一行代码是表格的列名，多个列使用”|”单竖线隔开，”||”双竖线区分输入和输出变量，即左边是输入值，右边是输出值
格式如下:
输入参数1 | 输入参数2 || 输出结果1 | 输出结果2

@Unroll注解，可以把每一次调用作为一个单独的测试用例运行，即把请求参数值和返回结果值的字符串里动态替换掉，”#uid、#postCodeResult、#telephoneResult” 井号后面的变量是在方法内部定义的，前面加上#号，实现占位符的功能。

spock其他符号的含义

//>>表示即模拟getUserInfo接口的返回结果
userDao.getUserInfo() >> [user1, user2]
//_表示匹配任意类型参数
List<UserDTO> users = userDao.getUserInfo(_)
//如果有同名的方法，使用as指定参数类型区分
List<UserDTO> users = userDao.getUserInfo(_ as String)

public KoreaCreditCardDiscount getKoreaCreditCardDiscount(String promotionId, RequestHeadDTO head, SearchResponseDTO responseDTO) {
    KoreaCreditCardDiscount koreaCreditCardDiscount = new KoreaCreditCardDiscount();
    if (StringUtils.isBlank(promotionId)) {
        return koreaCreditCardDiscount;
    }
    GetPromoteInfoRequestTypeDTO promoteInfoRequestTypeDTO = toPayDiscountForMerchantRequestDTO(head, responseDTO, promotionId);
    // 调用底层优惠券接口
    GetPromoteInfoResponseTypeDTO promoteInfo = getCreditCardPayWrapper.invoke(promoteInfoRequestTypeDTO);
    if (promoteInfo == null || CollectionUtils.isNullOrEmpty(promoteInfo.getPromotionList())) {
        return koreaCreditCardDiscount;
    }
 
    List<KoreaCreditCardDiscountDetail> koreaCreditCardDiscountDetailList = Lists.newArrayList();
    PromotionDTO promotion = promoteInfo.getPromotionList().get(0);
    KoreaCreditCardDiscountDetail discountDetail = new KoreaCreditCardDiscountDetail();
    discountDetail.setPromotionId(promotion.getPromotionId());
    discountDetail.setBrandId(promotion.getBrandIds());
    // 获取QConfig中优惠券品牌相关配置
    List<CreditCardBrandConfigDTO> creditCardBrandConfigList = creditCardBrandConfig.getCreditCardBrandConfig();
    if (CollectionUtils.isNotEmpty(creditCardBrandConfigList) && SourceEnum.ONLINE.equals(head.getSource())) {
        CreditCardBrandConfigDTO creditCardBrandConfig = creditCardBrandConfigList.stream()
                .filter(p -> StringUtils.equalsIgnoreCase(p.getBrandId(), promotion.getBrandIds())).findFirst().orElse(new CreditCardBrandConfigDTO());
        discountDetail.setBrandName(creditCardBrandConfig.getName());
    }
    discountDetail.setDeductionAmount(promotion.getDeductionAmount().doubleValue());
    discountDetail.setRuleType(promotion.getRuleType());
    koreaCreditCardDiscountDetailList.add(discountDetail);
    koreaCreditCardDiscount.setKoreaCreditCardDiscountDetailList(koreaCreditCardDiscountDetailList);
    koreaCreditCardDiscount.setDecimalCount(promotion.getDeductionAmount().intValue());
    return koreaCreditCardDiscount;
}

四、Void方法单测

无返回值方法的测试不能像上面几种有结果返回的，可以验证返回结果是否符合预期，
无返回值的方法主要是验证方法内部流转是否符合预期，比如应该走到哪个分支逻辑？
是否执行了这一行代码？for循环中的代码执行了几次？变量在方法内部的变化情况？
比如下面这个void方法的逻辑：

private void deductionDiscountedPriceInFlightList(FlightPolicySearchCommonResponseType responseType) {
    KoreaCreditCardDiscount koreaCreditCardDiscount = responseType.getKoreaCreditCardDiscount();
    if (koreaCreditCardDiscount == null || CollectionUtils.isNullOrEmpty(koreaCreditCardDiscount.getKoreaCreditCardDiscountDetailList())) {
        return;
    }
    for (PolicyInfoType policyInfoType : responseType.getPolicyInfoList()) {
        BigDecimal viewAvgPrice = policyInfoType.getPriceDetailInfo().getViewAvgPrice();
        viewAvgPrice = getCreditCardMapper.deductionMaxDiscountedPrices(viewAvgPrice, responseType.getKoreaCreditCardDiscount(), 0);
        // 其他业务逻辑省略... 比如转换对应的外币金额
        policyInfoType.getPriceDetailInfo().setViewAvgPrice(viewAvgPrice);
    }
}

测试代码：

def "测试deductionDiscountedPriceInFlightList()方法"() {

    then: "验证调用最大折扣计算方法的行为是否符合预期: 一共调用2次, 第一次输出折扣金额是100,第二次输出折扣金额是200"
    2 * getCreditCardMapper.deductionMaxDiscountedPrices(_, _, _) >> 100 >> 200
 
    and: "验证外币金额是否正确"

在then里通过“2 * getCreditCardMapper.deductionMaxDiscountedPrices(_, _, _)” 的方式，
验证deductionMaxDiscountedPrices()是否被调用了2次，前面的数字“2”表示这行代码执行了几次，
如果不是则测试失败，然后2次调用分别返回不同的值，第一次返回的折扣金额是100，第二次调用返回的折扣金额是200，再通过验证输入条件responseType的viewAvgPrice在for循环中的赋值是否符合预期情况，
这样的测试代码也更符合BDD行为驱动的思想。
另外如果要使用where的方式测试void方法需要注意，因为spock要求where里的表格至少有一个输入和一个输出，对于上面的代码如果要测试分支情况的话需要下面这种写法：

when:
method.invoke(searchMapper, responseType)
 
then:
responseType[0].priceDetailInfo.viewAvgPrice == price
 
where:
price            || _
BigDecimal.ONE   || _
BigDecimal.TEN   || _
BigDecimal.ZERO  || _

业务代码:

public GetPromoteInfoResponseTypeDTO invoke(GetPromoteInfoRequestTypeDTO requestDTO) {
    try {
        //GetCreditCardPayMapper的静态变量INSTANCE
        GetPromoteInfoRequestType request = GetCreditCardPayMapper.INSTANCE.toRequest(requestDTO);
        GetPromoteInfoResponseType response = getCreditCardHandler.invoke(request);
        GetPromoteInfoResponseTypeDTO responseTypeDTO = GetCreditCardPayMapper.INSTANCE.toReponse(response);
        if (Objects.nonNull(responseTypeDTO) && Optional.ofNullable(responseTypeDTO.getPromotionList()).map(List::size).orElse(0) > 1) {
            List<PromotionDTO> sortList = responseTypeDTO.getPromotionList().stream().sorted(Comparator.comparing(PromotionDTO::getDeductionAmount).reversed()).collect(Collectors.toList());
            responseTypeDTO.setPromotionList(sortList);
        }
        return responseTypeDTO;
    } catch (Exception e) {
        ELKLogUtils.addAppendedValue(StoredLogTag.RESPONSE_CONTENT, "getCreditCardActivity exception" + e.getMessage());
        return null;
    }
}

七、groovy闭包的用法

groovy里的闭包类似于Java中的lambda表达式（但比lambda要高级一些，比如它还有委托代理的功能）

语法如下：

//其中[closureParameters->]代表参数，多个参数用逗号分割，用->隔开参数与方法体，没有参数可以不写->
{ [closureParameters -> ] statements }

常用的用法如下：

{ printf 'Hello World' }   //打印 Hello World              
{ println it }             //闭包有默认参数it，可以不用申明                       
{ name -> println name }   // 定义一个name参数

考虑下面的业务场景：这个方法的功能是通过soaClient调用接口后获取返回数据，然后进行结果转换，得到response后再做其他的业务逻辑，但是调用的接口是非核心接口，报错不会影响主流程，所以通过try/catch处理，然后继续剩下的流程，伪代码如下：

public Response process(Request request) {
    Response response = null;
    try {
        SOAResponse soaResponse = soaClient.invoke(request); // 调用接口
        if (soaResponse != null) {
            response = new Response();
            response.setResultCode(soaResponse.isSuccess() ? 0 : -1);
            response.setResultMsg(soaResponse.getResultMsg());
        }
    } catch (SOAException ex) {
        logger.error(ex);
    }
 
    // 其他业务逻辑
    if(response != null) {
          if (response.getResultCode() == 0) {
                 
          }
          if (response.getResultCode() == 1) {
 
          }
          ...
    }
 
    return response;
}

正常我们测试这个方法，更多关注的是正常业务流程是否符合预期，比如说soaResponse返回结果是成功(success)还是失败，然后catch外代码在继续其他逻辑判断，但是有一种特殊情况就是"soaClient.invoke"接口抛异常了，比如TimeOutException，那么下面的代码"if (soaResponse != null) { "就不会执行了，这种情况如果我们不测试的话，catch外的代码可能会有问题，所以针对这种情况，单测最好也测试下，可以使用mock的方式让"soaClient.invoke"抛出一个异常出来，验证下流程是否严谨。如果希望能再where里既验证正常的业务逻辑又验证异常的逻辑，可以使用闭包的方式，如下：

@Unroll
def process() {
    given: "设置mock场景"
    soaClient.invoke(_) >> { soaResponse() } // { soaResponse.call() } 的简化写法
 
    when: "调用被测方法"
    def response = handler.process(new Request())
 
    then: "设置需要验证的属性"
    response?.resultCode == status // ?语法类似于.Net或kotlin的空指针安全，为null则不调用resultCode属性
 
    where: "验证不同业务场景"
    soaResponse        | status
    getSoaResponse(0)  | 0
    getSoaResponse(1)  | 1
    getSoaResponse(-1) | null // 这个case就是测试接口抛异常是否影响后续流程
}
 
def getSoaResponse(int index) {
    def soaResponse = {} // 声明一个闭包
    switch (index) {
        case 0:
            soaResponse = { new SOAResponse("success": true) }
            break
        case 1:
            soaResponse = { new SOAResponse("success": false) }
            break
        case -1:
            soaResponse = { throw new SOAException("接口异常") } // 如果不使用闭包的话，Spock编译到这里会直接抛出异常
            break
    }
    return soaResponse
}

getSoaResponse()方法返回的也是一个闭包，这样做的原因是因为”case:-1”是抛出一个SOAException异常，而不是正常的构造数据，如果不使用闭包的方式，Spock编译到这里会直接抛出异常，而"soaClient.invoke(_) >> { soaResponse() }" 这里加的花括号是表示只有在业务代码执行到"soaClient.invoke(_)" 方法时才执行闭包里的逻辑，所以这里使用了双层闭包。当然也可以针对这个异常case单独写一个单测，不放在原来的where里也可以，如果觉得闭包的语法不好理解的话。
